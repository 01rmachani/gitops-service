Branch Model

One repo. Infinite projects.

Each project gets its own isolated branch hierarchy, auto-bootstrapped on first push. No manual setup required.

main                     # service code only
├── proj-a-master        # auto-bootstrapped
│   └── proj-a-dev       # PR target
│       └── feat/add-auth
├── proj-b-master
│   └── proj-b-dev
│       └── feat/fix-bug

main only ever holds gitops-service code. Project code lives entirely in its own branch subtree — fully isolated.

1
Bootstrap master

Creates {project}-master from main with CI + review workflows committed.

2
Create dev branch

Creates {project}-dev from {project}-master as the integration target.

3
Feature branch per push

Each POST /push creates feat/<name|uuid> from {project}-dev.

4
PR opened automatically

Files committed and a PR opened targeting {project}-dev.

End-to-End Flow

Push code. Get a merged PR.

From a single HTTP call to a merged, AI-reviewed pull request — fully automated.

POST /push/sync
  │
  ├─ ensureProject(project)
  │    ├─ create {project}-master (if new)
  │    ├─ bootstrap .github/workflows/
  │    └─ create {project}-dev (if new)
  │
  ├─ create feat/<name|uuid> from dev
  ├─ push files from dir/
  └─ open PR → {project}-dev
       │
       GitHub Actions on {project}-master:
       ├─ ci.yml          → run tests
       ├─ code-review.yml → LLM diff review
       └─ auto-merge.yml  → squash & merge
⚡

Async or Sync

POST /push returns 202 immediately. POST /push/sync waits and returns the PR URL. Both use the same concurrency-capped queue.

🤖

AI Code Review

After CI passes, an LLM agent reviews the diff via OpenRouter and posts a structured comment. Outcome drives the auto-merge decision.

🔒

Path Traversal Protection

Every dir is validated to be inside INCOMING_DIR before any file is read. Traversal attempts return 400.

REST API

Simple HTTP interface.

All routes except GET /ping require an x-api-key header.

GET
/ping
Health check. Returns queue stats (active, queued, concurrency).
POST
/push
Fire-and-forget. Returns 202 immediately; branch + PR created in background.
POST
/push/sync
Waits for branch + PR creation. Returns full result including PR URL.
POST
/projects/:project/bootstrap
Manually trigger project bootstrap (idempotent).

Request body

{
  "project":     "proj-a",
  "dir":         "/mnt/incoming/proj-a/out",
  "description": "Add auth module",
  "feat_name":   "add-auth",
  "labels":      ["backend"],
  "source":      "my-service"
}

Response

{
  "project":   "proj-a",
  "branch":    "feat/add-auth",
  "pr_number": 42,
  "pr_url":    "https://github.com/..."
}
GitHub Actions

Four workflows. Zero manual steps.

Bootstrapped into every project branch automatically. Each has concurrency groups to cancel duplicate runs.

🧪

ci.yml

Runs project tests on every PR push. Must complete with success before code-review triggers.

🤖

code-review.yml

Triggers after CI passes (conclusion == 'success'). LLM reviews the diff via OpenRouter and posts a structured comment.

🔄

feat-auto-merge.yml

Triggers after code-review succeeds. Squash-merges if all gates pass, otherwise labels PR for human review with a reason comment.

📝

issue-to-branch.yml

Triggered by the gitops-push label on any issue. Parses the issue form body, calls POST /push/sync, and comments the PR link back.

Auto-merge Gates (all must pass)

✅Code Review workflow concluded with success
✅All CI checks green (no failure / cancelled / timed_out)
✅No human-review-required label on the PR
✅Repo variable AUTO_MERGE is not false
✅PR is MERGEABLE (no conflicts)

Concurrency & Dependabot

Every workflow has a concurrency group scoped to branch or issue number — duplicate runs are cancelled automatically.

Dependabot runs weekly PRs for both npm and github-actions versions. Zero configuration needed.

Setup Guide

Running in minutes.

Docker Compose is the recommended deployment. The service runs as a non-root user with a read-only volume mount.

1 — Clone & configure

git clone https://github.com/01rmachani/gitops-service
cp .env.example .env
# fill in GH_TOKEN, GH_OWNER, GH_REPO, API_KEY

2 — Run with Docker

docker compose up -d --build

3 — Push your first branch

curl -X POST http://localhost:3000/push/sync \
  -H "x-api-key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"project":"demo","dir":"/mnt/incoming/demo"}'

Required env vars

🔑GH_TOKEN — GitHub PAT with repo scope
📦GH_OWNER / GH_REPO — target repository
🔐API_KEY — shared secret for HTTP auth
📁INCOMING_DIR — host dir mounted read-only

4 — Create GitHub labels

gh label create "automated"              --color "0075ca"
gh label create "human-review-required" --color "e4e669"
gh label create "gitops-push"            --color "5319e7"

gitops-push triggers the issue-to-branch workflow.

GitHub repo secrets

🤖OPENROUTER_API_KEY — for AI code review
🔗GITOPS_API_KEY — for issue-to-branch workflow

Repo variables

⚙️AUTO_MERGE=false — disable auto-merge
🧠REVIEW_MODEL — override LLM model
🌐GITOPS_URL — service base URL for issue-to-branch
